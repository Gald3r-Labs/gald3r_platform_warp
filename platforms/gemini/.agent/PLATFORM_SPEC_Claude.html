What Makes This Platform Different

  • TWO competing hook config surfaces: .claude/hooks.json (gald3r) AND .claude/settings.json "hooks" (live Stop chat-logger). Ambiguous which wins; possible double-fire or no-fire.
  • Event-name casing mismatch: hooks.json lowercase vs official PascalCase (SessionStart/Stop). beforeShellExecution is a Cursor-era name with no Claude Code equivalent.
  • Mixed entry shapes inside a single hooks.json (flat vs nested-with-matcher).
  • local.settings.json non-standard filename present alongside official settings.local.json — likely dead/legacy.
  • Rules are plain .md (NOT .mdc); always-apply behavior leans on CLAUDE.md (@AGENTS.md import) rather than a per-rule activation engine.

Hook System

  • Type: json_config
  • Config file: .claude/settings.json key "hooks" (OFFICIAL, supported) — gald3r ALSO ships a top-level .claude/hooks.json (non-standard, partly-legacy; see §6/§9)
  • Events available (official Claude Code lifecycle): SessionStart, SessionEnd, UserPromptSubmit, PreToolUse, PostToolUse, Notification, Stop, SubagentStop, PreCompact
  • Event payload format: JSON (matcher-grouped: { "matcher", "hooks":[{ "type":"command", "command" }] })
  • Limitations: official shape is matcher-grouped nested arrays; flat { "command" } entries in hooks.json are suspect. _hook_md → additional_context injection (T1171) unconfirmed on Claude Code.
  • gald3r hook files: g-hk-*.ps1 invoked via powershell.exe -File .claude/hooks/g-hk-*.ps1; PreToolUse block (correct nested shape) is the most likely to fire; sessionStart/stop/beforeShellExecution are the suspect lowercase entries

Configuration Layout

.claude/
├── CLAUDE.md                ← gald3r-managed project instruction file (loaded every session); imports @AGENTS.md
├── settings.json            ← gald3r/user — permissions, env, mcpServers, hooks (official schema)
├── settings.local.json      ← user-local overrides (gitignored)
├── local.settings.json      ← legacy/local overrides (gitignored)  ❓ non-standard name — see §9
├── hooks.json               ← gald3r-managed hook wiring (NON-standard top-level file — see §6/§9)
├── rules/                   ← gald3r — always-apply modular rules (g-rl-*.md, plain .md)
│   └── g-rl-*.md
├── skills/                  ← gald3r — folder-per-skill (g-skl-*/SKILL.md); native Claude Code Agent Skills
│   └── g-skl-*/SKILL.md
├── agents/                  ← gald3r — subagent definitions (g-agnt-*.md); native Claude Code subagents
│   └── g-agnt-*.md
├── commands/                ← gald3r — slash commands (g-*.md → /g-*)
│   └── g-*.md
└── hooks/                   ← gald3r — hook scripts (g-hk-*.ps1) + companion g-hk-*.md (T1171)
    └── g-hk-*.ps1 / g-hk-*.md

gald3r Integration Notes

  • T1509 (this task's motivating bug): hook wiring on Claude Code MUST be done via the official settings.json "hooks" shape, NOT the lowercase top-level hooks.json. Migrate gald3r entries to { matcher, hooks:[{ type:"command", command }] } under settings.json and normalize event names to official PascalCase.
  • BUG-100 (resolved) confirms g-hk-session-start.ps1 IS exercised on some Claude Code versions, but does NOT confirm the lowercase hooks.json wiring is the firing path. Run a hook-firing smoke test before flipping Hooks to ✅.
  • When Claude Code updates, re-verify event names against https://docs.anthropic.com/en/docs/claude-code/hooks (breaking-change history to watch: hook config moved into settings.json).
  • Strengths over Cursor reference: native subagents (.claude/agents/) and native Agent Skills (.claude/skills/) require no shimming.
