# PLATFORM_SPEC — warp (Warp AI terminal / Agent Mode)

Authoring path: **UPDATE** existing `g-skl-platform-warp/SKILL.md`.

**Warp** is an AI-native terminal (not an IDE). Its AI surface in 2026 splits into **Active AI**
(always-on inline hints), **Agent Mode** (interactive multi-step task execution in the terminal),
and **Cloud Agents** (autonomous, event-triggered runs). It has **no file tree and no editor
pane** — its primitives are terminal sessions, project **Rules** files, and **Warp Drive** (a
cloud-backed store of Workflows, Notebooks, Prompts, and Environment Variables).

Crucially for gald3r parity, Warp **does** have a real, first-class project-rules mechanism
(`AGENTS.md` / `WARP.md` at the repo root, plus a UI-managed Rules feature) and **first-class MCP**
(it auto-discovers existing `~/.claude.json` / `.mcp.json` / `.codex/config.toml` server configs).
But it has **no `.mdc` rules-folder with glob scoping, no agent-definition file discovery, no
`SKILL.md` skill-discovery mechanism, and no lifecycle-hook system**. Workflows/Notebooks are
cloud-backed Warp Drive objects, not on-disk folder-per-skill primitives. This yields a mix of
genuine parity (rules, MCP) and honest hard gaps (skills, agents, hooks) — the correct factual
picture for a terminal-native platform, not an implementation failure.

This spec corrects stale assumptions in the prior SKILL.md, which claimed Warp has "no
project-level rules file" — that is **no longer true**: Warp picks up `AGENTS.md`/`WARP.md` at the
repo root as the direct equivalent of Cursor's rules / Claude Code's `CLAUDE.md`.

## 1. Folder Hierarchy

Warp is **not** folder-namespaced like Cursor (`.cursor/rules|skills|agents|commands|hooks`). There
is no `.warp/` project-config tree of gald3r primitives. Warp reads a small set of root-level rules
files and keeps most configurable objects in the cloud (Warp Drive) or in app settings:

```
<project-root>/
├── WARP.md                  ← project rules file (all-caps); takes priority if both exist — gald3r writes
├── AGENTS.md                ← Warp's default project rules file (generated by /init); shared w/ other agents — gald3r writes
└── <subdir>/AGENTS.md       ← per-subdirectory rules (most-specific wins) — optional

~/.warp/                     ← Warp's local app state (themes, launch configs) — platform-owned
~/.claude.json               ← MCP servers (auto-discovered by Warp) — shared, machine-specific
.mcp.json (root)             ← MCP servers (auto-discovered by Warp) — shared, gitignored per g-rl-02
.codex/config.toml           ← MCP servers (auto-discovered by Warp) — shared

(Warp Drive — CLOUD, not on disk)
├── Workflows                ← parameterized multi-step command sequences
├── Notebooks                ← runnable markdown docs
├── Prompts                  ← saved prompt snippets
└── Environment Variables    ← scoped env vars
```

- **gald3r writes**: `WARP.md` and/or `AGENTS.md` (project rules). Optionally seeds MCP into a
  shared config file Warp auto-discovers.
- **Warp owns**: the Rules-file recognition mechanism, Warp Drive cloud store, Agent/Active AI
  modes, app settings, and `~/.warp/` state.
- There is **no** `.warp/rules/`, `.warp/skills/`, `.warp/agents/`, `.warp/commands/`, or
  `.warp/hooks/`. gald3r primitives that depend on those directories have no native on-disk home.

## 2. AI Instruction File

- **Primary instruction surface: `WARP.md` / `AGENTS.md`** at the repo root. Warp auto-discovers
  both; the filename must be **ALL-CAPS** to be recognized. If both exist in the same directory,
  **`WARP.md` takes priority**. `AGENTS.md` is the default that `/init` generates (matching the
  cross-tool `AGENTS.md` convention), so `CLAUDE.md`/`GEMINI.md`-style instructions can be folded
  into the shared `AGENTS.md`.
- **Precedence** (most specific first): (1) the rules file in the current subdirectory, (2) the
  rules file in the repo root, (3) Global Rules (app-level, all projects). gald3r enforcement
  belongs in the **root `AGENTS.md` / `WARP.md`** so it applies project-wide.
- Project Rules apply **automatically** when an agent operates inside that project — no manual pin
  required (unlike aider, where `CONVENTIONS.md` must be added to `read:`).
- gald3r **generates/merges** `AGENTS.md` (and optionally `WARP.md`) during install.

## 3. Agents Support

- **No agent-definition-file discovery. ❌** Warp has no `g-agnt-*.md` scan, no roster of
  selectable gald3r agent files, and no per-agent journal directory mechanism.
- Warp *does* have **agent modes** — Active AI (always-on hints), Agent Mode (interactive
  multi-step), and Cloud Agents (autonomous, event-triggered on Warp's infrastructure). These are
  *operating modes of a single assistant*, not loadable gald3r agent definitions.
- gald3r's agent roster (task-manager, planner, qa-engineer, code-reviewer, …) cannot be selected
  as discrete agents. Agent *behaviors* can only be described inside `AGENTS.md`/`WARP.md` rules.

## 4. Skills Support

- **No `SKILL.md` skill-discovery / auto-load mechanism. ❌** Warp has no folder-per-skill scan and
  no "load skill when relevant" model-driven activation reading on-disk `SKILL.md` files.
- The **closest analogue is Warp Drive** — agents "pull directly from your Warp Drive contents
  (Workflows, Notebooks, Prompts, Environment Variables)" to inform responses. But Warp Drive is a
  **cloud-backed** store managed through Warp's UI/sync, not gald3r's on-disk `skills/g-skl-*/`
  tree. There is no install-time mechanism to register the gald3r skill set as Warp Drive objects
  from the repo.
- gald3r skills therefore have **no on-disk runtime home** on Warp. A skill's *instructions* reach
  Warp only if copied into `AGENTS.md`/`WARP.md` (which does not scale to the full skill set) or
  manually recreated as Warp Drive Notebooks/Workflows. This is a hard gap, not a config oversight.

## 5. Commands / Workflows

- **Warp Drive Workflows ⚠️** — Warp's command primitive is the **Workflow**: a parameterized,
  documented, multi-step command sequence stored in Warp Drive (cloud, version-controlled, team-
  shareable). These are the natural place for gald3r operational sequences
  (e.g. `gald3r status`, `gald3r commit T{id}`, `gald3r new-task`).
- **But Workflows are NOT a discoverable `@g-*`/`/g-*` command palette mapped from on-disk files.**
  gald3r ships `commands/g-*.md`; Warp has no `.warp/commands/` folder scan to install them as
  Workflows. Workflows must be authored/imported into Warp Drive separately. There is an **open
  feature request** for custom slash commands and agent hooks (warpdotdev/warp issue #6857) — not
  shipped as of this spec.
- Net: gald3r commands cannot be installed as native Warp commands from the repo; they map to
  manually-created Warp Drive Workflows, or are driven by typing the underlying shell invocation.

## 6. Hooks System

- **No lifecycle-event / hook system. ❌** Warp exposes no `sessionStart`, `stop`, `preToolUse`,
  `postToolUse`, or `beforeShellExecution` events, and no `hooks.json` / settings hook table for
  user-supplied scripts. Custom slash commands **and** agent hooks are an explicitly **open feature
  request** (warpdotdev/warp issue #6857), not a shipped capability.
- Consequence for gald3r: the PowerShell session-start / inbox-check / pre-commit hooks that
  auto-fire on Cursor and Claude Code **do not fire on Warp**. There is no wiring point.
- The nearest event-driven concept is **Cloud Agents** (autonomous, event-triggered runs on Warp's
  infrastructure) — but that is an agent-execution trigger on Warp's platform, **not** a hook bus
  that can run local gald3r `.ps1` scripts on session/tool lifecycle. A gald3r hook can only be run
  manually by invoking the script in the terminal.

## 7. Rules / Memory

- **Real native rules mechanism. ✅ (concept) / ⚠️ (format parity)** — This is Warp's strongest
  gald3r fit. Project Rules live in `AGENTS.md`/`WARP.md` and apply **automatically** within the
  project; Warp also has Global Rules (app-level, all projects) and a UI-managed Rules feature for
  reusable guidelines.
- **Format gap vs. Cursor**: Warp rules are plain markdown in a single root file (+ optional
  per-subdir files), **not** Cursor's `.mdc` files with `alwaysApply`/`globs:` frontmatter. There
  is no per-glob rule scoping mechanism beyond the directory the rules file sits in. So gald3r's
  fine-grained `g-rl-*.mdc` rule set must be **flattened into `AGENTS.md`/`WARP.md`** prose rather
  than installed as discrete scoped rule files. The persistent-memory *effect* is achieved (rules
  re-apply every session automatically); the *file model* differs.
- gald3r also keeps `.gald3r/learned-facts.md` for durable project facts, but Warp will only "see"
  them if they are referenced from `AGENTS.md`/`WARP.md`.

## 8. MCP Support

- **First-class. ✅** Warp natively supports the Model Context Protocol and **auto-discovers
  existing MCP server configs** from `~/.claude.json`, root `.mcp.json`, and `.codex/config.toml`,
  so the same server set is shared across Claude Code, Codex, and Warp without per-tool re-config.
- **Config**: MCP servers are managed through Warp's settings/UI and the auto-discovered shared
  config files above; there is no dedicated `.warp/mcp.json` required.
- For gald3r, MCP is the cleanest portability path: any MCP server gald3r relies on that is already
  in `.mcp.json` / `~/.claude.json` is reused by Warp automatically.
- ⚠️ The concrete active server set is per-machine and untested in CI; the *mechanism* is verified
  from docs, not from a live install in this environment.

## 9. Known Gaps vs. Cursor Reference

Per the decision tree in `g-skl-platform-cursor/SKILL.md` — a capability either (a) lives in common
`.gald3r_sys/`, (b) needs platform-specific config in `.gald3r_sys/platforms/.warp/`, or (c) is a
documented gap:

| Cursor-reference capability | Warp status | Disposition |
|---|---|---|
| `.cursor/rules/*.mdc` always-apply + glob rules | ⚠️ rules exist, but single-file `AGENTS.md`/`WARP.md` md, no `.mdc`/glob scoping | (b)/(c) flatten `g-rl-*` into root rules file; auto-applies per project |
| `agents/g-agnt-*.md` file discovery | ❌ no agent-file concept (only Active AI / Agent Mode / Cloud Agents modes) | (c) hard gap → behaviors described in `AGENTS.md` |
| `skills/g-skl-*/SKILL.md` discovery + auto-load | ❌ no on-disk skill mechanism; Warp Drive is cloud-only | (c) hard gap → no repo-installable runtime home |
| `commands/` `@g-*` palette from on-disk files | ⚠️ Warp Drive Workflows exist but not installed from repo; custom slash cmds = open RFC #6857 | (c) gap → manual Warp Drive Workflows |
| Lifecycle hooks (`hooks.json`, sessionStart, …) | ❌ none; agent hooks = open RFC #6857 | (c) hard gap → manual shell invocation; Cloud Agents ≠ local hook bus |
| MCP servers | ✅ first-class; auto-discovers `~/.claude.json` / `.mcp.json` / `.codex/config.toml` | (a)/(b) strong fit — shared MCP config reused |
| Project rules auto-apply | ✅ `AGENTS.md`/`WARP.md` auto-apply within project | (b) strong fit — the primary enforcement surface |

**Hard gaps (not achievable on Warp from the repo today): on-disk agents, on-disk skills,
extensible on-disk commands, lifecycle hooks.** The capabilities that map cleanly are
**project Rules** (`AGENTS.md`/`WARP.md`, auto-applied) and **MCP** (auto-discovered shared config).
Warp Drive Workflows/Notebooks are a *partial* analogue for commands/skills but are cloud-managed,
not repo-installable.
